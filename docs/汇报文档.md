# AI 差旅通（Travel Agent）— 多智能体差旅系统 项目成果汇报

> 一句话出差，全流程托管。

---

## 一、项目概述

**AI 差旅通** 是一个基于 LangGraph 的多智能体（Multi-Agent）企业差旅系统。用户只需输入一句自然语言，系统通过 **1 个 Orchestrator + 5 个专职 SubAgent** 协作，自动完成意图识别 → 查询改写 → 出差申请 → 审批跟踪 → 行程规划 → 预订执行的全流程闭环，并通过 SSE/WebSocket 实时向前端推送 Agent 的完整思考过程与结构化卡片交互。

## 三、核心技术亮点（10 大模块）

### 亮点 1：Multi-Agent 协作架构（Orchestrator）

- **Orchestrator + 5 SubAgent** 编排模式，基于 LangGraph StateGraph 构建有向状态图
- 6 个 Agent 角色按条件路由动态协作：IntentAgent → QueryRewriter → TravelApplyAgent → ItineraryAgent → BookingAgent → GeneralChat
- 条件路由 `_route_by_intent` 根据 `intent` 和 `intent_confidence` 智能分发，置信度 < 0.4 时降级为通用对话
- 支持流式（SSE / WebSocket）和同步（REST）两种管道执行模式
- 全局状态 `AgentState` 在各 Agent 间流转，包含意图、差旅上下文、工具结果、结构化数据等 15+ 字段

### 亮点 2：两阶段 NLU 管线

- **意图识别（IntentAgent）**：LLM + JSON 结构化输出，6 种意图分类（`travel_apply` / `itinerary_query` / `travel_status` / `booking` / `general_chat` / `unclear`），输出置信度 + 推理过程 + 实体抽取
- **查询改写（QueryRewriter）**：上下文补全 + 指代消解 + 实体标准化，将口语化问题转为结构化差旅查询，输出 `rewritten_query` / `resolved_entities` / `search_type`
- **实体合并策略**：`_merge_entities_to_context` 自动将 `destination`、`origin`、`start_date`、`end_date`、`reason` 写入 `TravelContext`，不覆盖已有非空字段

### 亮点 3：三级 Memory 记忆管理体系

```
┌──────────────────────────────────────────────────┐
│              Memory Manager 架构                  │
│                                                   │
│  短期记忆（Short-term）                            │
│  ├── 会话级存储，每个 session 独立                  │
│  ├── 容量上限 50 条，滑窗淘汰                      │
│  └── 支持按 limit 截取最近 N 条                    │
│                                                   │
│  长期记忆（Long-term）                             │
│  ├── 按 user_id 存储用户偏好                       │
│  ├── 跨会话持久化，支持偏好检索                     │
│  └── 驱动个性化出行推荐                            │
│                                                   │
│  工作记忆（Working Memory）                        │
│  ├── 按 session_id 存储任务中间状态                 │
│  ├── 记录当前申请进度、行程规划结果                  │
│  └── 为后续 Agent 提供即时上下文                    │
│                                                   │
│  上下文聚合                                        │
│  └── build_context_summary: 偏好+任务状态+最近对话  │
│      聚合为完整的 Agent 上下文摘要                   │
└──────────────────────────────────────────────────┘
```

- **滑窗淘汰**：短期记忆超过 50 条时自动裁剪
- **上下文聚合**：`build_context_summary()` 联合偏好、任务状态、最近 3 条对话，为 Agent 提供最全面的历史上下文
- **全局单例**：Singleton 模式确保所有 Agent 共享同一记忆实例

### 亮点 4：Context 上下文治理引擎

解决 LLM 上下文窗口有限 vs 对话历史无限增长的核心矛盾。

- **Token 预算管理**：单会话最大 8000 token，使用率 > 80% 自动触发压缩
- **智能估算**：中文约 1.5 token/字，英文约 0.25 token/字符，精准控制上下文大小
- **上下文压缩（`compact_context`）**：保留最近 6 条原始对话，早期消息提炼为摘要
- **差旅关键词提取**：`_summarize_messages` 自动提取目的地、出发地、日期、申请编号等差旅关键信息
- **相关性筛选（`select_relevant_context`）**：基于关键词重叠 + `recency_score` 时间衰减排序，返回最相关的 top_k 条上下文

### 亮点 5：可扩展 Skill 技能注册体系

参照 OpenClaw Gateway 的 Skill 机制，实现能力的插件化管理。

- **注册表模式**：`SkillRegistry` 管理所有可执行技能，支持动态注册 / 查询 / 执行
- **3 个内置技能**：

| 技能名 | 类别 | 说明 |
|--------|------|------|
| `full_travel_planning` | travel | 全量差旅规划（酒店+机票+高铁一站式） |
| `smart_hotel_recommend` | travel | 智能酒店推荐（评分+星级+距离综合排序） |
| `optimal_transport` | travel | 最优交通方案（并行查询机票+高铁，性价比排序） |

- **统一接口**：每个 Skill 遵循 `async (params: dict) → dict` 契约，TaskPlanner 可按名称调度任意 Skill
- **参数校验**：执行前自动校验 `required_params`，缺失时抛出 `SkillParamError`
- **可扩展性**：新增能力只需 `register()` 一行代码，无需修改 Agent 或路由逻辑

### 亮点 6：DAG 任务规划器（TaskPlanner）

参照 Manus 的 Task Planning 机制，实现任务的智能分解与调度。

- **预定义 DAG 模板**：`travel_apply`（3 节点）、`itinerary_query`（4 节点含依赖）
- **LLM 动态分解兜底**：模板未覆盖的意图由 LLM 在线生成任务分解方案
- **拓扑排序执行**：任务节点含 `depends_on` 依赖关系，`current_tasks` 返回无依赖且 PENDING 的任务
- **状态机管理**：每个任务节点有 `PENDING → RUNNING → COMPLETED / FAILED` 状态流转
- **前端实时推送**：每次状态变更通过 SSE / WebSocket 推送，前端 TaskPanel 实时渲染 DAG 进度

### 亮点 7：MCP 标准协议集成

- **双模式无缝切换**：`app_env == "development"` 且无 `MCP_AUTH_TOKEN` 时自动启用 Mock 模式；配置 Token 后直接切生产模式，**零代码改动**
- **JSON-RPC 2.0 协议**：通过 Higress MCP 网关统一调用钉钉差旅 API
- **可靠性保障**：tenacity 指数退避，最多 3 次重试
- **9 个 MCP 工具**：

| 工具 | 说明 |
|------|------|
| `travel_apply` | 提交出差申请 |
| `travel_apply_status` | 查询审批状态 |
| `get_hotel_list` | 查询酒店列表 |
| `get_hotel_detail` | 查询酒店详情 |
| `get_flights` | 查询航班 |
| `get_trains` | 查询高铁 |
| `book_hotel` | 预订酒店 |
| `book_flight` | 预订机票 |
| `book_train` | 预订高铁 |

### 亮点 8：双模型智能路由（ModelRouter）

- 智谱 **GLM-4** + 阿里百炼 **Qwen-Max** 双模型
- 按任务特性自动选择最优模型，温度参数针对任务类型精调：

| 任务类型 | 选择模型 | Temperature | 原因 |
|---------|---------|-------------|------|
| 意图识别 | Qwen | 0.1 | 分类任务需要稳定性 |
| 任务规划 | Qwen | 0.2 | 结构化输出准确性 |
| 对话生成 | Qwen | 0.7 | 中文流畅度与自然度 |

- 使用 LangChain `ChatOpenAI` 兼容接口，`_model_cache` 缓存实例避免重复创建

### 亮点 9：结构化卡片交互体系

前端 6 种富交互卡片，覆盖差旅全流程的结构化信息展示：

| 卡片类型 | 展示内容 | 交互能力 |
|---------|---------|---------|
| **ApprovalStatusCard** | 审批时间线、申请人、目的地、日期、事由 | 审批通过后显示「开始规划行程」按钮 |
| **FlightCard** | 航司、航班号、起降时间、价格、舱位、余座 | 一键预订 |
| **HotelCard** | 名称、星级、地址、价格、评分、设施、距离 | 一键预订 |
| **TrainCard** | 车次、起止时间、时长、价格、座位类型、余票 | 一键预订 |
| **BookingResultCard** | 订单号、状态（成功/失败/处理中）、详情、总价 | 状态展示 |
| **CardRenderer** | 路由分发器，根据 `cardType` 渲染对应卡片 | 自动匹配 |

### 亮点 10：流式交互与全程可观测

- **SSE 流式管道**：`POST /api/chat/stream`，前端 `useChat` 按行解析 SSE 事件
- **5 种事件类型**实时推送：

| 事件类型 | 说明 | 推送时机 |
|---------|------|---------|
| `agent_event` | Agent 思考过程（thinking/tool_call/message） | Agent 开始处理 / 工具调用 / 产出结论 |
| `structured_data` | 结构化卡片数据 | 审批状态、行程列表、预订结果 |
| `task_update` | 任务 DAG 状态 | 任务状态变更（plan / task 级别） |
| `token` | 流式文本片段 | LLM 逐 Token 输出 |
| `done` | 管道执行结束 | 所有 Agent 完成 |

- **双栏联动**：左侧 ChatPanel 展示对话 + 卡片，右侧 TaskPanel 实时渲染 DAG 进度和 Agent 活动日志
